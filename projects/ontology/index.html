Case Study

Manage a Scalable Ontology

Product DesignUX Research

Define, manage and deploy a set of classes for data annotation.

The Problem

Data Scientists working on the platform need a way to define and manage an ontology so that the included classes are available for data labeling and ultimately, model training.

Overview

An important part of getting data annotated is deciding what types of ‘things’ exist in the data. With respect to AI and Machine Learning, this collection of classes or categories is often called an ontology.

For example, an audio clip may get presented with an ontology of Speaker 1, Speaker 2, and Background Noise.

Along the way we found that ontology classes should include a title, a representative color and a unique identifier. The ontology itself may be short or long (in its number of classes), have nested classes, and/or be paired with predictive models.

A nested ontology architecture. Based on real customer data for a sports tracking model.
A ‘Persons’ smart class paired with a model might first predict persons in a string of text for a human annotator to review.

We set out to design the experience around creating, managing, and deploying a custom ontology that scales across many applications.

Users and Audience

  • Data Scientists
  • Machine Learning Project Managers
  • Internal Pro Service team members

My Role

I served as the lead designer on this project. My focus during this time was on Figure Eight’s entire suite of annotation tools. I designed these tools. Each one was designed to connect to this new ontology management experience. I worked closly with a Project Manager and both Front-End and Back-End Engineers to understand scope as the project progressed.

Length: 2 months with ongoing enhancements and usability improvements over 2 years.

Process

Initial Discovery

Sponsored Users

The team and I also worked closely with sponsored users; Select customers who could use and benefit from the solution we set out to create. By working with customers, we were able to understand the exact pain points of the specific problem and understand what value a new solution could bring.

Below are a few questions we discussed with these users. Many of these questions we returned to with each new iteration:

  • What is your current process?
  • What is painful about the experience?
  • What would be optimal/expected inputs and outputs?
  • (Based on X prototype) would it serve your needs?
  • (Based on X prototype) how might you be blocked from success?

Key Initial Learnings

  • Prior to this project, ontologies were handled with customer created CSV files that were created offline and submitted to customer support via email. The support team would manually upload the files into an AWS bucket and pull them into the customers account with Javascript. The process was confusing, lengthy, and prone ot errors.
  • The information that is essential to a basic ontology is class title, representative color, and an output value.
  • Interestingly users often struggle to even define what classes to include. Much work gets done to decide on ontology classes before the ontology is ready create and use in the platform.

Iterative Prototyping

I next began multiple rounds of prototyping with increasing levels of fidelity. They were put in front of our sponsored users and other stakeholders for feedback. After each round I would incorporate feedback, providing variations and presenting artifacts as needed.

A Note on Interactions

One of the most impactful interaction points of the handoff design was the ability to drag and drop to reorder and place classes. This interaction scaled well for when we introduced the idea that ontology classes could be nested within each other.

Initially we allowed users to edit a class right in the list view, which helped with speed and efficiency. Though a solid interaction, we found that this wouldn’t scale. We anticipated that an ontology class would have much more information attached to it than what would be shown in the at-a-glance listing table. Ultimately, class editing was made to happen in a modal, a good decision given that rich HTML descriptions, predictive model pairing, and metadata were soon to get implemented.

A Note on Visual Design

The visual design leaned heavily on our design system and existing patterns. New, thoughtful styling and treatments were created to handle the drag and drop functionality, class expand/contract functionality, and the pairing of a color (including the color picker).

Deep Dive on Smart Ontologies

As an example of an ontology use case, a data scientist might ask annotators to find all the ‘Dates’ present in text-based financial records.

Locating dates in text is not a unique endeavour and predictive models exist to identify dates in text generally. A predictive model could take a first pass to locate all dates and have the annotators simply double check and correct those predictions, thus saving human effort and cost.

What if you could create an ontology class that pairs with a model to predict instances of that class, before an annotator reviews, corrects, and generally fills in the gaps? After finding evidence that predictions do save time, we set out to solve this problem.

While creating or editing a class, the user can select pre-existing predictive models to run on the data before the annotator begins.
Before pairing predictive models, the user will want to test how well it works and if it is a good option for them to include.

Outcomes: A Scalable and Influential Offering

At first, this implementation was used as an ontology manager for just one annotation tool. Over the next two years the experience grew to include the following enhancements:

  • Manage Ontologies for Pixel-level annotation (Initial Offering)
  • Introduce image shapes ontology
  • Introduce video shapes ontology with multiple instances of each class.
  • Include text annotation ontology. Introduce rich HTML descriptions.
  • Add in the ability to nest classes.
  • Add the ability to pair predictive models across multiple data types.
  • Add the ability to associate meta-data with each class.

Ultimately, we met each of these challenges. All said and done, the most rewarding aspect of building this experience was its ability to build off of and scale the initial offering. As new customer pain-points were identified, we found many solutions could be built off of the existing experience, validating the designs scalability and its influence on the company's roadmap.
