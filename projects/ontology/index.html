Manage a Scalable Ontology

An important part of gathering training data is deciding what types of ‘things’ exist in the data. With respect to AI and Machine Learning, this collection of classes is called an ontology.

For those modeling emergency response calls, raw audio clips may get presented with an ontology of Speaker 1, Speaker 2, and Background Noise. An annotator would then label the audio clips accordingly.

How can a data scientist effectively manage and deploy an ontology for data annotation?

A preview of the UX solution.

The Problem

As a data scientist, I need a way to create and manage a list of 'things' that exist in my data so that others can properly label my data for model training.

I was the lead product designer on the project. In addition to UI/UX and visual design I conducted research and user testing. Much of my focus during this time was on the businesses entire suite of annotation tools. I designed these tools and each one was designed to connect to this new ontology management experience.

The ontology for this audio clip is Speaker 1, Speaker 2 and Background Noise.

Initial Discovery

The team and I worked closely with sponsored users: select customers who could use and benefit from the solution we set out to create. By working with them, we were able to understand the exact pain points of the problem and understand what value a new solution could bring.

Below are a few questions we discussed with these users. Many of these questions we returned to with each new iteration.

  • What is your current process?
  • What is painful about the experience?
  • What would be the optimal/expected inputs and outputs?
  • (Based on X prototype) would it serve your needs?
  • (Based on X prototype) how might you be blocked from success?

Below are some of our initial learnings, gathered from sponsored users.

  • Prior to this project, ontologies were handled with customer created CSV files that were created offline and submitted to customer support via email. The support team would manually upload the files into an AWS bucket and pull them into the customers account with Javascript. The process was confusing, lengthy, and error prone.
  • The information that is essential to a basic ontology is class title, representative color, and an output value.
  • Users often struggle to even define what classes to include. Much work gets done to decide on ontology classes before the ontology is ready to use in the platform.

The as-is user journey created for ontology development.

Iterative Prototyping

I conducted multiple rounds of prototyping with increasing levels of fidelity. Designs were put in front of our sponsored users and other stakeholders for feedback. After each round I would incorporate feedback, providing variations and presenting artifacts as needed.

Lo-Fi to Hi-Fi

We wanted to nail down the information architecture considerations of the new experience. Where would ontologies be managed? Initially, the most optimal location was deemed out of scope as there would need to be major rearrangement of existing components.

We settled on an alternative location but made design decisions that would allow for the experience to be 'relocated' in the future shoudl we invest more in this offering.

Interactions

One of the most impactful interaction points of the handoff design was the ability to drag and drop to reorder and place classes. This interaction scaled well for when we introduced the idea that ontology classes could be nested within each other.

Initially we allowed users to edit a class right in the list view, which helped with speed and efficiency. Though a solid interaction, we found that this wouldn’t scale. We anticipated that an ontology class would have much more information attached to it than what would be shown in the at-a-glance listing table. Ultimately, class editing was made to happen in a modal, a good decision given that rich HTML descriptions, predictive model pairing, and metadata were soon to get implemented.

Visual Design

The visual design leaned heavily on our design system and existing patterns. New, thoughtful styling and treatments were created to handle the drag and drop functionality, class expand/contract functionality and the pairing of a color (including the color picker).

Deep Dive on Smart Ontologies

As an example of an ontology use case, a data scientist might ask annotators to find all the ‘Dates’ present in text-based financial records.

Locating dates in text is not a unique endeavour and predictive models exist to identify dates in text generally. A predictive model could take a first pass to locate all dates. The annotator can then simply double check and correct those predictions, thus saving human effort and cost.

What if you could create an ontology class that pairs with a model to predict instances of that class, before an annotator reviews, corrects, and generally fills in the gaps? After finding evidence that predictions do save time, we set out to solve this problem.

The user can pair predictive models to run on the data before annotation begins.

Before pairing a predictive model, users want to see how well it performs.

Outcomes

At first, this implementation was used as an ontology manager for just one annotation tool. Over the next two years the experience grew to include the following enhancements:

  • Manage Ontologies for Pixel-level annotation (Initial Offering)
  • Introduce image shapes ontology
  • Introduce video shapes ontology with multiple instances of each class.
  • Include text annotation ontology. Introduce rich HTML descriptions.
  • Add in the ability to nest classes.
  • Add the ability to pair predictive models across multiple data types.
  • Add the ability to associate meta-data with each class.

Ultimately, we met each of these challenges. All said and done, the most rewarding aspect of building this experience was its ability to build off of and scale the initial offering. As new customer pain-points were identified solutions could be built off of the existing experience, validating the designs scalability and its influence on the company's roadmap.
