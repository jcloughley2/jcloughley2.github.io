Welcoming Meta developers to their day

VS Code is the most used tool by Meta's community of several thousand developers and machine learning scientists. Though it's one piece of a larger developer ecosystem, many engineers start their day in the application.

Though VS Code is a Microsoft product, Meta builds and maintains Meta-specific experiences through extensions, including for this new experience.

A preview of the UX solution.

The Problem

As a Meta developer, I would like to gain context and jump into my work (code authoring) quickly and with as little friction as possible.

My team was tasked with helping engineers to jumpstart developer work. In order to start, a few things are important, including context about what they plan to work on. This may include information on their previous work. Understanding this, they next need the avenues to set up their workspace. This includes a connection to the server they will be working on as code changes are made remotely.

For the adoption of new features and a trending positive sentiment towards the tool, we also wanted to present product release notes.

Project Setup

This project took the span of a half. I worked closely with 3 engineers invested in making this solution come to life. Design artifacts were produced using Figma and Figjam.

Initial Discovery

The biggest area of understanding was around how developers plan for their day and choose what to work on. Initial discovery for this project was found using two means.

Customer Feedback Roundtables

We put together 3 customer roundtables with 3 - 6 participants each. Participants were users or customers of VS Code. We asked about all that they did before they started code authoring in VS Code and we asked about their workflows for starting to write code. We dug into the specific movements within the tool and the pain points they encountered. We asked what would speed this process up for them. If they had a magic wand, what would they change?

Doing this in roundtables allowed for discussion with multiple viewpoints. It ultimately helped us identify specific problems to solve as well as their prioritization.

We followed up with a few of these participants during the iterative design process to make sure we remained on the right track.

Developer Journey Map

Even before this project began, myself and a group of other designers and researchers created the Meta Developer Journey Map, the purpose of which was to understand our entire ecosystem of developer tools and how developers move through them. Our methodology was to combine and analyze anonymized data on tool usage and tool transitions with qualitative data gathered from video diaries of users completing their work.

We leaned on the findings from this map to design the new experience. Having this information gave us clarity into the actions, shortcuts, and information sources a user looks at before starting to code. We took some of these things and moved them into this single start-of-the-day experience.

A snapshot of the developer journey map artifact.

Iterative Prototyping

We entered roughly 10 weeks of iterative design prototyping. I met weekly with stakeholders to walk through the latest designs, collect feedback and answer any questions. This time was also used to communicate with engineers about the implementation and technical feasibility.

Concurrently, each week, I would meet with users to make sure we were on track from their perspective.

The implementation was first launched to a small group of our overall users. We kept on the lookout for any feedback or bugs coming in related to the changes. This information helped direct design decisions and next steps as well.

Early lo-fi iteration of a new homepage experience.

Deepdive on setting up a workspace

Our discovery exposed all the different ways developers set up their workspace. It entails a few considerations:

  • What type of connection environment is most efficient? Some developers need a server that spans the entire repo of the company. Others need a server that is tailored to a segment of the entire repo. Others work locally and don't access the repo at all.
  • What am I working on today? Context on the day's work is important. Some developers set up in the same way each day because they work fairly consistently on the same files. Some developers start each day differently, depending on the task they need to complete, the bug they need to squash, or the project space they are working in.
  • Am I continuing to work on the same thing as yesterday? This consideration opens the door for an option to set up just as the user left it the day before.

Because of these considerations, developers need the flexibility of many options to set up their workspace. We identified the key pathways, including:

  • Start workspace from a blank connection environment.

    We discovered that more often than not, developers do access consistently from the same pool of connection environments so a key interaction pattern was surfacing recent connection environments, with the option to find one in the general listing. Note there are 4 different types of connection environments.
  • Start workspace from previous work.
  • Start workspace from a project.

An early iteration of workspace setup pathways.

Product Release Notes Learning

We initially thought to present this information directly on the homepage, but we spoke to users and found it had low viewership. Asking for more detail, we learned that as homepage content it gets perceived as static. There is no notification for when content changes and the motivation to manually look at this every once in a while wasn't quite there yet for our developer community.

Our first approach to showing product release notes.

We switched gears and developed a more familiar pattern for release notes. When there is an update, it is shown once to the user in a separate tab when the tool is opened. The approach is a bit more forceful but at a time when the information is relevant. We retained the ability to find these notes at any time, should the user choose to.

The selected approach to showing product release notes.

Outcomes

The new experience provides quick workspace setup and addresses much of the pain our users expressed about this process. Time to start a server reservation became 40% faster than with the previous experience.

An important part of this new experience is the ability to quickly reserve a recently used connection environment type. These are used 20% more than reserving one from the general listing, validating 'recents' as useful.

Product release notes get 2x impressions compared to a posting (internal advertisement). This increases product sentiment and drives adoption of new features.

I saw part of this effort in action, and was amazed! Thank you for having the prior context saved and accessible like that! I also had been using the [connection environments] website to connect, but now that connection shows up in 'recents' - A Meta Data Engineer
